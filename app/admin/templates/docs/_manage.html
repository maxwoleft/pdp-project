{# Керування Сценаріями та Профілями #}

Сценарії (Scenarios)

Що це

Сценарій — це заготовлений тестовий діалог з очікуваною поведінкою AI. Використовується для QA-тестування: чи AI коректно реагує на типові запити клієнтів.

Кожен сценарій містить:

  • • Slug — унікальний ID
  • • Description — короткий опис задачі
  • • Country — країна салону (ua/pl/gb)
  • • Turns[] — список повідомлень клієнта (по одному за раз)
  • • Expectations[] — що повинна містити відповідь AI (ключові слова/факти)
  • • Must_not[] — чого НЕ повинно бути (наприклад, вигадані послуги)
  • • Tags[] — категоризація для filter
  • • Preset_salon_id — конкретний салон у якому виконувати тест

Як використовувати вручну

  1. 1. Перейти у розділ Scenarios
  2. 2. Натиснути "+ Новий сценарій".
  3. 3. Заповнити поля. У Turns вписати реальні фрази клієнта (по одній за рядок).
  4. 4. У Expectations вказати що повинна містити фінальна відповідь AI (наприклад: "корекція брів", "грн", "вільний час").
  5. 5. Зберегти. Натиснути "Run" щоб запустити сценарій проти живого AI.
  6. 6. Переглянути результат: pass/fail + повну розшифровку діалогу + які expectations спрацювали або провалились.

Коли створювати новий сценарій

  • • Користувач поскаржився що AI неправильно зрозумів запит — створи сценарій з тим запитом.
  • • Запустили нову послугу/бренд — додай сценарій "клієнт питає про <бренд>".
  • • Перед deploy нової версії — прогнати всі сценарії як smoke-test.
  • • Регулярно (раз на тиждень) — щоб впіймати regression від змін у CRM/контенті.

Профілі послуг (Service Profiles)

Що це

Профіль — концептуальне групування послуг однієї теми (наприклад "Манікюр", "Фарбування Balmain", "Подологія"). Один профіль обʼєднує усі канонічні ключі (canonical_keys), що належать до цього концепту.

Profile несе expert-контент для AI: на які проблеми відповідає, кому підходить, які переваги, ключові слова для пошуку. AI використовує це для семантичного матчингу запитів клієнтів до правильної групи послуг.

Структура профілю

  • Name — людська назва ("Манікюр", "Фарбування La Biosthetique"). Бачить адмін.
  • Canonical key — primary ID профілю (унікальний у межах country).
  • Canonical keys[] — список ckeys що входять у profile. Кожен ckey представляє конкретну реалізацію послуги (різні tier, brand, довжини, тощо).
  • Key descriptions — короткий intent-context per ckey, для AI: "коли клієнт згадує X — пропонувати цей ckey".
  • Translation (per language, default uk):
    • — Short description: 1-2 речення про profile
    • — Addresses problems: список проблем що вирішує (для всіх ckeys profile сумарно)
    • — Target audience: кому підходить
    • — Benefits: що отримає клієнт
    • — Keywords: синоніми + niche-tokens (для search)
    • — Cross-sell: що порадити разом
    • — Embedding: vector усього контенту, для search_by_concern
  • Ckey overrides — опціональний JSON у translation: для ckeys що мають специфічні concerns відмінні від profile defaults (наприклад "фарбування коренів" має свою проблему "відросли корені" окремо від загального "Фарбування").

Як редагувати вручну

  1. 1. Перейти у Service Profiles.
  2. 2. Знайти profile через пошук або список. Натиснути на name → відкривається detail.
  3. 3. Угорі — основні поля профілю + кнопка Edit translation для редагування addresses_problems / target_audience / keywords / etc.
  4. 4. Внизу detail — список Канонічні ключі + послуги. Натиснути на ckey щоб розкрити panel "Контент для AI пошуку".
  5. 5. Заповнити поля специфічні саме для цього ckey (якщо відрізняються від profile defaults). Зберегти.

Коли треба втручатися вручну

  • • AI не знаходить послугу при типовому запиті → додати missing keywords у profile keywords.
  • • AI пропонує неправильний variant у профілі → додати specific overrides на конкретний ckey.
  • • Зʼявилися дві profiles з однаковим змістом (через зміну CRM) → merge через скрипт merge_profiles.py або вручну.
  • • Profile містить чужі ckeys (наприклад "Чоловічі послуги" має жіночий ckey) → видалити неправильний з canonical_keys[] вручну.
  • • Sales pitch застарів → редагувати у translation (хоча зараз sales_pitch прибрано, бо однаковий для всіх variants не релевантний).

Що відбувається автоматично

  • • Auto-linking: нові ckeys прив'язуються до profiles за brand + token-overlap + scope guards.
  • • LLM auto-fill: profiles без контенту автоматично заповнюються через OpenAI gpt-5-mini після кожного синку.
  • • Keyword enrichment: distinctive токени з ckey-sample names auto-додаються у profile.keywords.
  • • Re-embed: при зміні контенту translation embedding оновлюється.

Взаємодія Сценаріїв та Профілів

Сценарії тестують чи правильно AI використовує дані Профілів. Workflow:

  1. 1. Клієнт скаржиться "AI не зрозумів запит про X" → створюємо сценарій з цим запитом.
  2. 2. Запускаємо сценарій → fails → дивимось чому AI промахнувся.
  3. 3. Знаходимо що у Профілі бракує ключового слова / неправильні concerns.
  4. 4. Редагуємо Профіль → re-embed.
  5. 5. Повторно запускаємо сценарій → passes.
  6. 6. Сценарій залишається у наборі як regression test для майбутніх змін.
