{# Опис технології — повний опис системи #}

Що це таке

Beauty AI Chat — інтелектуальний помічник для салонів краси. Клієнт описує свою потребу природньою мовою (українська, російська, англійська, польська), а AI підбирає конкретну послугу, майстра та вільний час у салоні. Підтримує текст, голосові повідомлення та фото.

1. Тип бота

Це AI-agent з функціями/інструментами + RAG-компонентом для пошуку знань.

Гібридна архітектура:

  • • AI-agent — LLM рішає які tools викликати на основі контексту розмови (не скриптовий потік).
  • • Function calling — LLM має 10 tools, кожен — функція що звертається до БД або CRM API (повний список нижче).
  • • RAG для пошуку — vector similarity search над embedding-ами послуг і концептів. Результати tool calls передаються назад у LLM як контекст.
  • • НЕ скриптовий — немає predefined flow tree. LLM сам визначає послідовність дій.
  • • НЕ intent-based — немає окремого intent-classifier. Намір розпізнається через семантичний пошук та reasoning LLM.

1.1 Повний список tools (функцій LLM)

LLM викликає одну або кілька функцій за turn. Кожна повертає JSON з даних БД/CRM.

list_salons
Повертає список салонів обраної країни. Викликається на початку розмови, щоб клієнт обрав салон. Можна фільтрувати за містом.
list_categories
Повертає категорії послуг конкретного салону. Без parent_id — кореневі. З parent_id — підкатегорії. Допомагає клієнту orientуватися у каталозі.
search_services
Шукає послуги салону за вільним запитом і/або категорією. Vector search по service.name_embedding. Використовується коли клієнт описує що хоче напряму ("манікюр з покриттям", "фарбування волосся"). Фільтри: brand, volume_ml, zones, session_minutes, ampules.
search_by_concern
Шукає послуги за описом проблеми або бажаного результату ("хочу щоб волосся сяяло", "часто облазить лак", "груба шкіра на стопах", "не хочу нарощування але хочу довгі вії"). Vector search по profile.translation.embedding — знаходить profile-кандидати з їх services + concerns.
find_masters_for_service
Повертає список майстрів салону що виконують конкретну послугу (за service_id). Включає imʼя, посаду, рейтинг.
get_available_slots
Повертає вільні часові вікна для послуги. Якщо master_id не вказано — шукає по всіх майстрах які виконують цю послугу. Дати у ISO форматі. Дані live з AIHelps API (TTL 45 сек у Redis).
find_service_and_slots
Комбінований tool — пошук послуг + майстрів + вільних слотів за один виклик. Заміна 3 окремих tools (search_services + find_masters + get_available_slots). Префернтний для booking flow коли клієнт описав і потребу/послугу, і дату ("хочу записатися на стрижку завтра після обіду").
create_booking
Створює запис у CRM після підтвердження клієнтом. Обовʼязково спочатку отримати імʼя і телефон. slot_ids беруться з результату get_available_slots. Після виклику: AIHelps API booking + Telegram нотифікація + email клієнту.
get_client_bookings
Повертає активні (майбутні) бронювання клієнта за номером телефону. Використовується коли клієнт хоче дізнатися "куди я записаний" або перед скасуванням.
cancel_booking
Скасовує запис і звільняє слоти у CRM. Перед викликом — обовʼязкове підтвердження клієнта. Сповіщає менеджера салону про скасування через Telegram.

Усі tools зареєстровані у app/agents/tools/builder.py. Кешуються у Redis (TTL 10 хв для каталогу, 45 сек для слотів).

2. AI-моделі та платформа

Conversation (текст)
OpenAI GPT-5 — primary. Fallback GPT-5-mini через hybrid_router.
Голос (voice → text)
OpenAI Whisper whisper-1 — транскрипція аудіо (webm/ogg/mp3, до 25MB).
Зображення (image → text)
OpenAI Vision GPT-5 з image_url input — аналізує фото клієнта (стан волосся, нігтів, проблема шкіри), повертає опис.
Vector search
OpenAI text-embedding-3-small (1536d, cosine similarity). Embeddings зберігаються у PostgreSQL через pgvector.
Query rewriter + LLM reranker
OpenAI GPT-4o-mini — швидкий, дешевий (~200-400ms на виклик).
Cron auto-fill (batch контент)
OpenAI GPT-5 + GPT-5-mini — генерація key_descriptions, profile-level concerns, translations.

3. Логіка прийняття рішень

LLM (GPT-5) приймає рішення на основі:

  1. 1. System prompt — інструкції у app/agents/instructions/:
    • — golden_rules.md: правила безпеки і anti-hallucination
    • — tools_reference.md: коли який tool викликати
    • — booking_flow.md: етапи запису
    • — persona.md + communication_style.md: стиль майстра-сенсея
    • — sales_dna.md: продажні принципи
    • — country_overrides/: специфіка по країні (UA/PL/GB)
  2. 2. Контекст розмови — попередні turns зі сесії (Redis), salon_id, country, language.
  3. 3. Останнє повідомлення клієнта + опціонально transcribed voice / image description.
  4. 4. Tool calls — LLM вирішує що потрібно: пошук послуги? слотів? майстрів? Викликає tool → отримує JSON → синтезує відповідь.
  5. 5. Multi-turn loop — LLM може зробити кілька tool calls в один turn (наприклад: search_services → find_masters → get_slots → синтез у одну відповідь).

4. Визначення потреби клієнта

  • • Semantic reasoning у LLM — GPT-5 з system prompt розуміє намір тексту.
  • • search_by_concern tool — клієнт описує проблему ("пігментація", "відросли корені") → vector cosine match по profile.translation.embedding (включає addresses_problems, target_audience, keywords).
  • • search_services tool — клієнт називає послугу напряму ("хочу манікюр") → vector match по service.name_embedding.
  • • Voice/image — додаткові модальності транскрибуються/аналізуються, текст іде у той же reasoning loop.

5. Підбір категорії / послуги

Дворівневий пошук:

Рівень 1 — profile (концепт): AI бачить top-K profiles з найбільшим cosine match. Profile агрегує всі ckeys (canonical_keys) певної теми.
Рівень 2 — конкретний service: з profile → запит services з salon_id + matched ckeys → отримує реальні service.id з ціною, тривалістю, brand-ом. LLM обирає один за service.name який найкраще відповідає запиту.

LLM-reranker (gpt-4o-mini) опціонально передоцінює top-K candidates для +5-10% accuracy на неоднозначних запитах.

6. Робота з профілями послуг

Профіль — концептуальне групування послуг ("Манікюр", "Фарбування Balmain", "Подологія"). Містить:

  • • canonical_keys[] — список ckeys (нормалізовані ID послуг)
  • • translation per language — addresses_problems, target_audience, benefits, keywords, cross_sell, embedding
  • • ckey_overrides — опціональний JSON: специфічні concerns для конкретного ckey (наприклад "корені" має свою проблему окремо від "повного фарбування")

AI шукає profile через embedding (search_by_concern) → profile дає набір services з salon → LLM обирає specific service. Per-ckey overrides додають точність коли variants мають різні concerns.

7. База знань

Так, бот використовує базу знань. Зберігається у PostgreSQL на DigitalOcean droplet:

booking.salons — глобально, country-aware
<country>.service — каталог послуг (ua, pl, gb)
<country>.category — CRM-категорії
<country>.employee — майстри з графіком
public.service_profile — концептуальні групи
public.service_profile_translation — expert-контент + embedding

Інструкції для LLM (правила, стиль, flow) — Markdown-файли у app/agents/instructions/, завантажуються при старті у system prompt.

8. Створення запису

Бот може самостійно створювати запис у CRM — НЕ лише консультує.

Через tool create_appointment:

  1. 1. Збирає від клієнта: послуга + майстер + час + телефон + ім'я.
  2. 2. Викликає AIHelps API → бронює слот.
  3. 3. Відправляє Telegram-сповіщення менеджеру салону.
  4. 4. Відправляє email-підтвердження клієнту (через SMTP).
  5. 5. Запис відображається у CRM салону одразу.

Інфраструктура

1. Де розміщений чат

DigitalOcean droplet (Ubuntu 24.04, 2 vCPU / 2 GB RAM, Frankfurt) → https://ai-chat.p-de-p.com. Stack: FastAPI + uvicorn (2 workers) під systemd (aichat.service). Nginx + Let's Encrypt SSL.

2. Де розміщена база знань

PostgreSQL 16 на тому ж droplet. Per-country schemas + pgvector extension. Інструкції LLM — у коді репозиторію (app/agents/instructions/*.md).

3. Де розміщені профілі послуг

PostgreSQL: public.service_profile + public.service_profile_translation (+ service_profile_variant, service_profile_version для версій).

4. Де зберігаються діалоги з клієнтами

Redis на droplet — активні розмови (ConversationCache), TTL = ~24 год. Кожен діалог: history з ConversationTurn (role + content + has_image). Логи дублюються у /var/log/aichat/app.log для аналітики.

5. Як чат підключений до самозапису

Окремий сервіс pdpbooking (booking_london_predict, Flask на тому ж droplet, https://booking.p-de-p.com). У booking-застосунку є CTA-банер що веде у AI чат через /chat/start?salon_id={uuid} — клієнт одразу потрапляє у чат свого салону. AI чат бронює слоти у тій самій CRM що використовує booking.

6. Як чат отримує інформацію про салони, послуги, майстрів і вільний час

Двома способами:

  • • Кешований каталог у PG (послуги, майстри, категорії) — синхронізується з AIHelps CRM cron-ом двічі на день (3:00 та 13:00).
  • • Live AIHelps API виклики — вільні слоти запитуються у реальному часі через get_available_slots tool (TTL кешу 45 секунд у Redis).

7. Як відбувається створення запису

Tool create_appointment: збір даних → AIHelps API POST /appointments → Telegram bot нотифікація менеджеру → SMTP email клієнту → запис у локальну booking.appointmentsleads для аудиту.

8. API та інтеграції

  • • OpenAI API — chat completions (GPT-5/5-mini/4o-mini), embeddings, Whisper, Vision
  • • AIHelps CRM API — каталог послуг, майстри, слоти, створення запису (per salon database_code)
  • • Telegram Bot API — повідомлення менеджеру про новий запис (per salon bot/chat)
  • • SMTP — email-підтвердження (Gmail / per salon)
  • • Mailchimp — опц., маркетингові розсилки
  • • EasyTip — інтеграція платежів (через booking, не AI чат напряму)
  • • Redis — внутрішній API кешу tools

Так, проект підтримуваний. Що потрібно:

  • • Стандартний Python/FastAPI стек з типовими патернами (SQLAlchemy 2.0, async/await, Jinja2).
  • • Структурований код: app/api/, app/agents/, app/adapters/, app/domain/, app/infrastructure/.
  • • Cron pipeline задокументований у /etc/cron.d/aichat-sync на droplet.
  • • Конфіг через env vars (app/core/config.py).
  • • Документація — цей розділ + README у репо + Markdown інструкції у app/agents/instructions/.
  • • Скрипти-утиліти у scripts/ для типових операцій (sync, auto_link, fill content, тести).
  • • Тести: scripts/test_chat_* — e2e сценарії з anti-hallucination.
  • • Що передати новому розробнику: GitHub доступ + .env креди + SSH ключ + DNS + AIHelps/OpenAI/Telegram credentials.

Cron pipeline

Автоматизація 3:00 та 13:00:

3:00 — sync_from_crm: повна синхронізація послуг/майстрів зі усіх AIHelps салонів
3:15 — cleanup_addon_categories: чистка add-on/modifier категорій
3:18 — cleanup_addon_services: чистка add-on послуг за name pattern
3:25 — auto_link_missing_keys: linking ckeys → profiles (brand-aware + scope guards)
3:27 — link_by_category: fallback linking за category-name
3:28 — llm_fill_key_descriptions: AI intent-detection описи per ckey
3:30 — embed_translations: оновлення translation.embedding
3:32 — llm_fill_new_overrides: AI заповнення expert-контенту нових profiles
3:34 — enrich_profile_keywords: auto-додавання niche-tokens з ckey samples
3:40 — compute_profile_salons: оновлення profile.salon_ids/cities
13:00 — light midday repeat (sync + auto_link + enrich)

Безпека та anti-hallucination

  • • System prompt у golden_rules.md явно забороняє вигадування цін, назв послуг, імен майстрів, вільних слотів.
  • • tools_reference.md: "tools — єдине джерело правди про послуги, майстрів, час".
  • • Порожній результат пошуку → AI каже клієнту "на жаль, такої послуги немає" і пропонує альтернативу.
  • • Per-language reply detection — AI відповідає тією мовою якою клієнт пише (uk/ru/en/pl).
  • • Multi-turn memory — Redis зберігає попередній контекст в межах сесії.
  • • Anti-hallucination tests — scripts/test_chat_* з NEGATIVE scenarios (вигадані послуги → AI має чесно сказати "немає").
